## Hypothesis
Muon computes the polar factor of the gradient — "closest orthogonal matrix to G."
Cayley computes the Stiefel tangent direction using the WEIGHT matrix:
"what rotation of the current weights best follows the gradient?"

These answer fundamentally different questions.

## Architecture
Momentum → Cayley Direction (uses W) → Optional NS cleanup → Update

## Key Formula
```
A = momentum @ W.T - W @ momentum.T  (skew-symmetric generator)
direction = A @ W                      (Stiefel tangent vector)
orth = NS(direction, steps=3)          (cleanup + normalization)
```

Efficient for tall matrices (m > n):
```
direction = momentum @ (W.T @ W) - W @ (momentum.T @ W)
```

## Why This Is Different
- Muon uses ONLY the gradient. The weight matrix is irrelevant to the direction.
- Cayley uses BOTH gradient AND weights. The direction adapts to the current weight structure.
- Cayley direction = 0 when gradient aligns with weights (no rotation needed).
- Cayley direction is large when gradient is misaligned (large rotation needed).
- This is manifold-aware optimization: rotational geometry instead of Euclidean.

## Cost
Cayley computation ≈ 1 NS step equivalent. With 3 NS cleanup steps = 4 total ≈ 20% cheaper than Muon.

## Analysis
- Catastrophic failure — 0.32 worse than Muon
- The Cayley direction (A@W where A = u@W.T - W@u.T) is fundamentally wrong for this task
- When gradient aligns with weights, Cayley direction → 0 (no rotation needed). This happens frequently.
- The direction encodes "rotation of weights" rather than "steepest descent direction"
- Even with NS(3) cleanup, the underlying direction quality is poor
- NS(3) can't fix a bad direction — it can only orthogonalize what it's given
- Loss is still improving at step 2000 (converging, just very slowly) — suggests the direction has SOME signal, just much less than polar factor

## Verdict
FAILED. The Stiefel tangent direction is wrong for optimization — Muon's polar factor captures gradient information much better than the rotational component.
