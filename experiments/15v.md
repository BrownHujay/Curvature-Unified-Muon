# Series 15: Universal Muon — Kill the AdamW Crutch

**Files:** `cum/tensor/universal_muon.py`
**Status:** PENDING

## Motivation

Muon only orthogonalizes 2D hidden-layer weights (~60-70% of params). Everything else (embeddings, biases, LayerNorm) gets AdamW. This creates a Frankenstein optimizer where two parameter groups have completely different convergence dynamics sharing one training loop. Adam's slow adaptive warmup means embeddings/attention are at different optimization stages than hidden layers at any given step.

**Goal:** Single optimizer for ALL parameters. Uniform convergence behavior. No AdamW anywhere.

## Phase 1: Universal Muon (Simple)

Apply appropriate orthogonalization-like operation per parameter shape:

| Shape | Example | Operation |
|-------|---------|-----------|
| 2D (m×n), min(m,n)>1 | All weights, embeddings | NS orthogonalization (same as Muon) |
| 1D (d,) | Biases, LN scale/shift | Unit normalization: v/‖v‖ |

Key design decisions:
- **Embeddings get NS too.** They're 2D. No reason to exclude them.
- **1D gets normalized.** The "polar factor" of a vector is just its direction.
- **Single LR, single momentum** across all params.
- **Aspect ratio scaling** per param: sqrt(max(1, m/n)) for 2D, configurable for 1D.
- Start with basic Muon (no blending). Add combined/TD(λ) recipe in Phase 2 if Phase 1 shows promise.

## Key Hypothesis

Unified convergence dynamics > split optimizer, even if individual operations are suboptimal per group. The synchronization benefit (all params in same optimization phase) outweighs any loss from applying NS to embeddings or normalization to biases.

## Concerns
- Embeddings have different spectral structure (rare tokens → near-zero rows). NS might handle this poorly.
- LN scale params are initialized to 1.0 and barely move. Normalizing their gradient might over-amplify updates.
- Weight-tied head/embedding means the same param gets gradients from both LM head loss and embedding lookup — NS might struggle with this dual role.
- 1D scaling factor unclear — sqrt(d) from aspect ratio could be too aggressive.
